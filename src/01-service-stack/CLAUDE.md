# The Service Stack

At the top are the people. At the bottom is storage. Everything in between is the engine.

The simplest possible architecture that delivers Family Command Center outcomes through native text messaging. No apps to install. No channels to manage. Just one messaging address the family can message, optimized for phone-native texting. A separate local-only admin UI exists for operator configuration — it is not part of the messaging experience.

Every family member has the assistant saved as a contact on their PHONE. The assistant has one shared messaging identity and no name.

## System Promises

The assistant never sends more than it should. Outbound is budgeted, batched, and routed to the narrowest audience.

The assistant never acts without permission on sends, financial actions, or system changes.

The assistant always responds where you are. Proactive messages go to the narrowest thread that fits.

The assistant shifts tone and behavior by topic while remaining one coherent identity.

Every source of information flows through the same queue, same worker, same dispatch logic.

The design is testable. Every behavior maps to a scenario that can be generated, evaluated, and used to refine the model.

One messaging identity. One queue. One set of rules. Infinite scenarios to prove it.

## Scenario Testing Framework

The system must support infinite scenario generation and evaluation against the model's rules. Scenarios are the test harness for the design, not just the software.

Every scenario has five parts:

- **Trigger** — what starts it (human message, scheduled event, email, calendar change, or combination).
- **Expected dispatch** — who gets what, in which thread, at what priority, in what tone, and when.
- **Expected input handling** — what responses might come and what each means.
- **Escalation check** — if expected responses don't arrive, does the right escalation happen?
- **Collision check** — does this scenario's outbound conflict with other active scenarios?

Scenarios are generated by combining a random topic, trigger type, participant set, and complication (no response, conflicting events, multi-step, rescheduled, ambiguous input, concurrent scenarios). Failures refine the model — the topic behavior profiles, routing rules, escalation paths, and batching thresholds.

### Example Scenario

Trigger: An adult entity messages "vet appointment for PET next Tuesday at 10am" in the broadest shared thread.

Expected dispatch: Confirmation in the originating thread immediately, using the calendar topic's precise and logistical tone. Reminder to the responsible adult in their private thread Monday evening, batched priority. Follow-up to the responsible adult in their private thread Tuesday at noon asking for vet notes, using the pets topic's caretaker tone.

Expected input handling: The responsible adult replies "done, got shots updated too" — mark appointment complete, log health note for PET under the pets topic, confirm the addition. A child entity sends a positive reaction on a chore reminder — mark chore complete, log completion time.

Escalation check: If the responsible adult doesn't respond to the vet follow-up by Tuesday 6pm, it appears in their evening digest as an open item. No thread escalation — pets is a low-escalation topic.

Collision check: If the vet follow-up, a chore reminder, and a bill alert are all due Tuesday afternoon for the same entity, are they batched into one message or spaced appropriately? Does the immediate-priority bill alert go out separately while the others batch?

When a scenario produces a wrong outcome — wrong thread, wrong timing, wrong tone, missed escalation, unnecessary notification — it's not a code bug. It's a model refinement. The topic behavior profile needs adjusting, or the routing rules need a new qualifier, or the batching logic needs a new threshold.
