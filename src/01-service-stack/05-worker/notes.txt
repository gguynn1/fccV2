Required tools/dependencies:
- Node.js (LTS) with TypeScript (strict, ESM) for all runtime modules.
- `src/types.ts` for shared enums and cross-service vocabulary.
- Respect service and type boundaries: no direct runtime imports from sibling services.
- Local `types.ts` in this folder for scope-owned contracts.
- An orchestration layer that processes one queue item at a time.
- Service client interfaces for topic profile, routing, budget, escalation, confirmation, and state.
- Decision objects that capture topic, audience, priority, and final action.

Recommended tools/dependencies:
- `tsc --noEmit` to verify type boundaries.
- `eslint` to keep imports and unused symbols clean.
- `prettier` to keep docs and source formatting consistent.
- `vitest` for fixture-driven verification where the scope has behavior worth testing.
- Scenario tests covering the fixed processing sequence end to end.
- Decision tracing so worker outcomes can be inspected when behavior changes.

Technologies and integrations:
- The worker is a strict TypeScript orchestration layer over queue, supporting-service, and action-router contracts.
- pino logging is the right fit for per-step decision traces and auditability.
- Vitest scenario suites with mocked service clients are the clearest way to verify the decision sequence.
- Stale catch-up: on startup after downtime, the worker must check each item's age and relevance before processing — items past their window (e.g., "pickup in 30 minutes" from 3 hours ago) are logged silently, never dispatched late.
- Network outage: if a service call fails due to network (Claude API, Twilio), the item should be retried via BullMQ backoff, not dropped.
